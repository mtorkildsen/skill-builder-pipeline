Orchestrator — Pipeline Manager
Wraps the entire pipeline. Detects entry point, plans sessions, tracks state, verifies files before and after each stage, generates handoff prompts, and launches subagents with user approval. Does not write SKILL.md files — it manages the pipeline around the stages that do.
Initialize
Read Pipeline References
Read references/pipeline-directory-map.md and references/skill-writing-patterns.md before doing anything else. These define path conventions and quality standards.
Entry Detection
3 Entry Points
  • New skill — idea from scratch. Collect name + root, assess complexity (Low/Medium/High), generate build plan, initialize pipeline-state.json
  • Resume — pick up midstream. Read state file, verify output files for completed stages, resume from current stage
  • Tweak — small change. Classify as patch (direct edit), minor revision (Stage 3 re-run), or major revision (Stage 1 re-run)
State Tracking
pipeline-state.json
Tracks skill name, root, current stage, completed stages, stage outputs, version, model recommendation, and entry type. Updated after every stage completes.
Stored At
{skill-root}/pipeline-state.json
Verification
Pre/Post Stage File Checks
Before launching any stage, verify required inputs exist. After completion, verify outputs exist before updating state.

Subagent Protocol
Announce Before Launch — Never Run Silently
Stages 2 (Prep) and 4 (Release) run as subagents. Before launching:
  • Tell the user what the subagent will do, what it reads, what it produces
  • Ask for explicit approval
  • STOP. Do not launch until confirmed.
Briefing templates in references/subagent-briefings.md. Failures reported to user with retry/manual/abort options.
Failure Routing
QA Failure → Targeted Build-Sesh Handoff
When manual QA fails, generates a targeted handoff prompt listing only the failed items. Tracks cycle count in state file.
⚠ After 3 failure cycles, escalate: recommend revisiting Stage 1 (Design) before another revision

Design — Planning Sesh Interview
Interactive · Opus recommended
Turn a vague idea into a concrete, unambiguous design document through guided Q&A. One question at a time, with options and tradeoffs at every turn.
Entry
Capture the Rough Idea
User provides anything from a single sentence to a rough draft. Claude confirms understanding in 1–2 sentences.
Initialize
Create Design Doc Artifact with 12-Item Checklist
A live artifact is created with 12 checklist items, each marked ⬜ (open). Items already answered from user input are marked ✅.
1. What is it
2. Actor & goal
3. Entry point
4. Trigger / detection
5. Main workflow steps
6. User gates
7. Plugins & artifacts
8. Output
9. Ambiguous steps
10. Edge cases
11. Open questions
12. Blockers
Core Loop
Guided Interview — One Question at a Time
For each unresolved checklist item, Claude asks a question with:
  • Why it matters — what goes wrong if left ambiguous
  • A/B/C options — concrete choices with tradeoffs
  • Recommendations when the user is stuck
Branches are surfaced and the user steers. Items get parked as Open Questions or Blockers if unresolvable.
Repeats until all 12 items resolved or parked

Prep — Handoff Sesh
Subagent · Sonnet recommended
Transform the design doc into 4 build-ready deliverables so build-sesh produces high-quality output on the first iteration without guessing. Progressive save after each deliverable — if the session crashes, completed work is on disk.
Setup
Read Design Doc & Flag Open Items
Read the full design document. Surface any Open Questions or Blockers for the user to resolve or explicitly defer before proceeding.
Deliverable 1
Register Classification
Single pass through every instruction in the design doc, asking three questions per item:
  • Q1: Is the designer's intent clear? — resolves design ambiguity
  • Q2: Is there only one way to implement? — writes explicit decision logic
  • Q3: Procedure or Principle? — classifies register (rigid steps vs. flexible guidelines)
Plus Edge Case Routing: each edge case goes to one of three destinations:
  • Into the skill as a principle (reveals non-obvious design intent)
  • Into test cases (most go here — prove it works through testing)
  • Dropped (defensive speculation that never occurs in practice)
Level 3 Gate
STOP — User Approves Register Classification
User reviews the full analysis. After approval:
Saved To
handoff-sesh/register-classification.md
Deliverable 2
Reference File Research
Identifies domain knowledge gaps that would cause inconsistent outputs. Uses best-practices-researcher skill to create each reference file (in parallel when possible). Structured for quick lookup — tables and headers, not prose. Skipped if no domain knowledge is needed.
Saved To
handoff-sesh/references/{descriptive-name}.md
Deliverable 3
Test Cases with Mock Files
3–5 realistic test scenarios covering:
  • Happy path — standard workflow
  • Friction path — non-standard inputs at every step (pick unexpected options, partial answers, nonstandard file paths)
  • Edge cases — pulled from the routing table in Deliverable 1
Each includes: descriptive name, full mock file specs, user parameters, and specific verifiable expected outputs.
Level 3 Gate
STOP — User Approves Test Cases
Saved To
handoff-sesh/test-cases.md
Deliverable 4
Verification Checklist
Per-step failure mode catalogue. Read templates/verification-checklist-template.md before generating. For each design doc step, populate:
  • Gate compliance — current strength level + escalation path
  • Step skipping risk — whether Claude might decide this step isn't needed
  • Step merging risk — distinct steps that might get combined
  • File hallucination — verify every file-producing step actually saves
  • Reference drift — verify reference files are actually read
  • Output format deviation — verify format requirements
  • Artifact launch compliance — verify visual outputs launch correctly
Plus cross-cutting checks: happy path completion, friction path handling, session recovery, file inventory.
Level 3 Gate
STOP — User Approves Verification Checklist

3
Build — Build Sesh v2.0
Interactive · Opus recommended
Write the SKILL.md, run the embedded linter, test with subagents, gather human feedback via eval viewer, improve, and repeat. Progressive save after every revision — if the session crashes, the last revision is on disk.
Read First
Load Handoff-Sesh Deliverables
Before writing or revising, read all 4 deliverables in order:
  • handoff-sesh/register-classification.md — how to write each instruction
  • handoff-sesh/references/ — domain knowledge files
  • handoff-sesh/test-cases.md — use these instead of generating your own
  • handoff-sesh/verification-checklist.md — failure modes to watch for
If handoff-sesh/ doesn't exist (skill built outside the pipeline), proceed with standard create-from-scratch workflow.
Step A
Capture Intent
Confirm what the skill does, when it triggers, expected output format, and whether test cases are needed. Much of this is already answered by the handoff package.
Step B
Write the SKILL.md Draft
Produces the skill file with:
  • Frontmatter — name, description (appropriately "pushy" for reliable triggering)
  • Body — instructions under ~500 lines, using register classifications from handoff
  • Bundled resources — scripts/, references/, assets/ as needed
Principles explain the why; procedures are explicit steps. Explain reasoning rather than relying on rigid MUSTs.
Step C
Save → Lint → Auto-Fix → Save
Progressive save SKILL.md to disk immediately. Then run the 9-check linter against skill-writing-patterns.md and the exemplar skills:
1. Gate language strength
2. Progressive disclosure (<500 lines)
3. Instruction voice (directive)
4. File output specificity
5. Weak language detection
6. Artifact launch method
7. Description field triggers
8. Reference file usage
9. Exemplar structural comparison
Auto-fix unambiguous issues: weak gates before file writes → Level 3, "refer to" → explicit Read, descriptive voice → directive, missing verification → add it. Save again after auto-fixes. Judgment-call issues go in linter report for user.
Step D
Run Test Cases
Use test cases from handoff-sesh/test-cases.md. For each test case, spawn subagents:
  • First iteration only — baseline comparison (with-skill vs. without-skill) to confirm the skill adds value
  • Subsequent iterations — with-skill only (user can request another baseline anytime)
While runs execute, draft quantitative assertions. Capture timing data from each completion notification.
Step E
Grade & Launch Eval Viewer
Grade each run against assertions (scripted checks preferred). Launch the eval viewer (generate_review.py --static in headless environments). User reviews outputs + linter report in the viewer, leaves feedback per test case. Cross-reference against verification-checklist.md while reviewing.
Step F
Improve the Skill (Anti-Degradation Triage)
Each issue is classified before touching the skill:
  • Knowledge gap — add a narrow rule with reasoning
  • Judgment gap — rewrite instruction to explain why and when
  • Behavioral regression — reconcile conflicting constraints into one principle
Constraint count stays stable across iterations. If it's growing, pause and consolidate.
Steps B–F repeat: Write → Save → Lint → Fix → Save → Test → Review

Release — QA Sesh
Subagent + Manual QA · Sonnet recommended
The release pipeline. Enforces versioning, documents changes, generates a manual QA template as .html, and packages into a distributable .skill file. 6 steps, 3 gates.
Responsibility boundary: QA Sesh ships skills — it does not fix them. The only writes to SKILL.md are metadata (version bump in the metadata block). If something fails, findings are reported for build-sesh to address.
Step 1
Intake
Locate the skill directory, validate SKILL.md exists, parse frontmatter (name, description, metadata.version). Accept the revision summary as pasted text or file path. If summary is too vague, ask user to elaborate or fall back to auto-diffing.
Step 2
Version Bump
Read current version from metadata.version in YAML frontmatter. Propose:
  • Patch (x.x.+1) — Bug fixes, typos, formatting
  • Minor (x.+1.0) — New features, new steps, expanded capabilities
  • Major (+1.0.0) — Breaking changes, removed steps, restructured flow
First-ever build initializes at 1.0.0. Version stored in metadata block, not top-level frontmatter.
Step 3
Changelog
Create or update CHANGELOG.md at the skill root using Keep a Changelog categories (Added, Changed, Fixed, Removed, Deprecated, Security). Newest entries prepended. Confirm file exists after writing.
Gate 1 — Level 3
Version + Changelog Confirmation
Present proposed version, change type classification with reasoning, and formatted changelog entry. User can confirm, override version level, edit wording, or reject. After confirmation: write version to SKILL.md frontmatter, write CHANGELOG.md, create qa-sesh/v{version}/.
Step 4
Manual QA Template (.html)
Read handoff-sesh/verification-checklist.md before generating. Cross-reference against SKILL.md to produce test cases covering every step's failure modes. The .docx template includes:
  • Header: skill name, date, version, model used
  • Test cases organized by workflow step (numbered: 1-1, 1-2, 2-1...)
  • PASS / FAIL / N/A columns with colored cell backgrounds
  • Predetermined user answers for happy path — exact inputs at each gate
  • Predetermined user answers for friction path — non-standard inputs at each step
  • Cross-cutting checks: happy path completion, friction path handling, session recovery, file inventory
Present using present_files tool so user can open directly.
Saved To
qa-sesh/v{version}/qa-template.html
Gate 2 — Level 3
Manual QA Pass / Fail
User tests the skill against the QA template.
  • Pass → proceed to packaging
  • Fail → log findings, route back to Stage 3 (Build Sesh) with specific failure details
⚠ After 3 consecutive failure cycles (patch bumps addressing similar issues), escalate: recommend revisiting Stage 1 design
Step 5
Package
Run package_skill.py from the build-sesh-v11 sibling skill directory. Verify the .skill file was created successfully.
Step 6
Archive
Create version archive zip containing all files in the versioned folder. Confirm archive exists.
Saved To
qa-sesh/v{version}/{skill-name}-v{version}-qa.zip
Gate 3 — Level 3
Final Summary
Present: version number, all artifacts with paths, .skill package location, archive location. Present .skill file using present_files tool.
Final Output Structure
{skill-root}/ SKILL.md (version bumped in metadata block) CHANGELOG.md (new entry prepended) qa-sesh/ v{version}/ qa-template.html (manual QA with PASS/FAIL/N/A) {skill-name}-v{version}-qa.zip (version archive) {skill-name}.skill (distributable package)
